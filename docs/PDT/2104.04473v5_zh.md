## 摘要

大规模语言模型已在多项任务上取得最好结果，但高效训练面临两大挑战：
1. **显存受限**：单卡、甚至单台多卡服务器都装不下最新的大模型；
2. **算力受限**：训练所需的浮点运算量大到让端到端训练时间变得不现实。

为此，研究界提出了 **张量并行（tensor parallelism）** 和 **流水线并行（pipeline parallelism）** 等模型并行方案，但若直接把它们堆在上千张 GPU 上运行，扩展性会迅速劣化。

本文给出一种把 **张量并行、流水线并行与数据并行** 三者合理组合（作者称之为 **PTD‑P**）的方法，使训练可线性扩展到数千张 GPU。主要贡献有：

- 提出一种新颖的 **交错式（interleaved）流水线调度**，在显存占用与既有方案相当的前提下，将吞吐再提升 10% 以上；
- 在 3072 张 A100 GPU 上，对一个万亿参数的 GPT 模型达到 **502 PFLOP/s** 的聚合吞吐，**单卡 52%** 的理论峰值利用率；
- 系统性分析了多种并行维度的相互作用，给出超参（尤其是 micro‑batch 大小）选择的分析模型与经验法则。

> 图 1（见 `figs/page-01.png` 右上角）给出了 SOTA NLP 模型的参数规模随时间近乎指数增长的趋势：从 GPT(110M) → BERT(340M) → Turing‑NLG(17B) → Megatron‑LM(8.3B) → GPT‑3(175B) → ELMo 等系列。

## 1. 引言

基于 Transformer 的语言模型（BERT、GPT‑2/3、RoBERTa、T5、XLNet 等）在 NLP 领域取得了突破性效果，并被证明是强大的 zero/few‑shot 学习器。它们的下游应用包括：客户反馈摘要、自动对话、语义搜索、代码自动补全等。

然而，训练此类模型存在两方面困难：
- **装不下**：即使 NVIDIA 最新的 80GB A100，也无法装下完整模型参数；
- **训不动**：例如 1750 亿参数的 GPT‑3，若单卡 V100，需要约 288 年才能完成一次训练。

这迫使我们必须引入并行化。数据并行虽然好用，但存在两点局限：
1. 超过某一规模后，单卡的 batch 变得很小，GPU 利用率下降、通信开销上升；
2. 最大可用设备数被 batch size 所限。

**模型并行的两种主流范式：**

- **张量并行（intra‑layer / tensor MP）**：把 Transformer 层内的矩阵乘切分到多张 GPU 上。对 20B 参数以下的模型，在 DGX A100（8×80GB）内部效果良好，但更大的模型需要跨节点，跨节点的 all‑reduce 会跑到远慢于 NVLink 的 InfiniBand 上；过高的张量并行度还会把 GEMM 切得太小，使 GPU 计算利用率降低。
- **流水线并行（inter‑layer / pipeline MP）**：把模型按层切分并放到多张 GPU 上，一个 batch 被拆为若干 micro‑batch，通过流水化提高设备利用率。为了保持严格的优化器语义（同步权重更新），每个 batch 结束时必须进行 **流水线刷新（pipeline flush）**，此时设备空闲，形成 **气泡（bubble）**。气泡最高可吃掉 50% 的训练时间，除非显著增大 batch 使 micro‑batch 数远大于流水线深度。

**本文要回答的核心问题：**

> *"在给定 batch 大小并保持严格优化器语义下，如何组合这些并行手段才能最大化大模型的训练吞吐？"*

作者的答案是 **PTD‑P**：节点内使用张量并行、跨节点使用流水线并行、再叠加数据并行扩展。在优化良好的集群上（节点内 NVLink、节点间高带宽 IB）该方案接近线性扩展到 3072 张 A100，**单卡 163 TFLOP/s**，聚合 **502 PFLOP/s**，对应一次端到端训练约 **3 个月**。

同时对比了 ZeRO‑3，在 175B 与 530B 模型上 PTD‑P 吞吐高出约 70%（因跨节点通信更少）。

**文章给出的几条指导原则（takeaway）**：
- 不同并行维度的相互作用非线性、影响气泡与通信开销；次优的（张量，流水）组合可导致吞吐下降 2×。
- 流水线调度策略影响通信量、气泡大小与激活显存；**交错式调度** 最多可提升吞吐 10%。
- **micro‑batch 大小** 同时影响显存、kernel 算术强度与气泡大小，最优值与问题相关，调整可带来约 15% 吞吐提升。
- 在规模化训练下，通信密集：万亿模型在 3072 卡上流水通信的有效二分带宽需要 892 GB/s、数据并行 13 TB/s；更慢的互连或更通信密集的切分都会拖慢扩展。
- 作者未做自动搜索（不同于 FlexFlow、PipeDream 等），而是给出一套在实践中好用的启发式（见第 3 节）。

## 2. 并行模式

本节讨论用于突破单卡显存瓶颈的并行技术。**PTD‑P** 是张量并行（T）、流水线并行（P）和数据并行（D）的组合。下图示意了张量并行与流水线并行的复合：

### 2.1 数据并行

经典做法：每个 worker 各持一份完整模型，数据按 shard 分配，worker 周期性 all‑reduce 梯度以保持参数一致。对装不下单卡的模型，可改为在切片后的模型上再叠加数据并行。

### 2.2 流水线并行

Transformer 各层被切分到多台设备。由于本文讨论的模型是层间结构相同的 Transformer 堆叠，可以均匀地把层分到各个 pipeline stage。非对称结构的最优分层交给 FlexFlow、PipeDream 等工作。

一个 batch 被切成若干 micro‑batch，然后在设备间做流水。为了让前后传播看到一致的参数版本，我们采用 **同步权重更新**，因此需要在每个 batch 末尾引入 **刷新（flush）**，保持优化器严格语义；代价是产生 **pipeline bubble**（气泡）。PipeMare、PipeDream‑2BW 等工作通过 **异步 / bounded‑staleness** 去掉了 flush，但放宽了权重更新语义，不在本文讨论范围。

#### 2.2.1 默认调度：GPipe 与 PipeDream‑Flush

**GPipe**（见图 3）先对一个 batch 内所有 micro‑batch 依次做前向，再做反向，最后 flush 并更新参数。

记号：
- $m$：一个 batch 中 micro‑batch 数；
- $p$：流水线级数（流水线并行度）；
- $t_f, t_b$：单 micro‑batch 前向、反向时间；
- $t_{id} = m\cdot (t_f + t_b)$：理想 batch 时间。

GPipe 的气泡总时间为

$$t_{pb} = (p - 1)\cdot(t_f + t_b)$$

气泡占比为

$$\mathrm{BubbleFrac} = \frac{t_{pb}}{t_{id}} = \frac{p - 1}{m}$$

要让占比小，就必须 $m \gg p$，但这样就得为 $m$ 份 micro‑batch 的中间激活留出显存（即使启用了激活重算，也至少要存每个 stage 的输入激活）。

**PipeDream‑Flush**（也称为 1F1B，见图 4 上半部分）通过 warm‑up 阶段让各设备先做不同数量的前向，再进入 1F1B 稳态——每做一次前向就跟一次反向。1F1B 将 **在飞 micro‑batch 数** 限制到流水深度 $p$（而非 $m$）；气泡大小不变，但 **激活显存** 从与 $m$ 成比例下降到与 $p$ 成比例，在 $m \gg p$ 时显著节省显存。

#### 2.2.2 交错式（Interleaved）调度——本文新贡献

为了进一步压缩气泡，作者让每张设备承担 **多段不连续的层**（称为 "model chunk"），即每个设备同时担任多个流水阶段：例如原本 device 1 负责层 1–4，device 2 负责 5–8……改成 device 1 负责 {1,2,9,10}、device 2 负责 {3,4,11,12}……每段 chunk 的计算量变小。

在此基础上对 1F1B 做改造（见图 4 下半部分），要求一个 batch 中的 micro‑batch 数必须是流水线并行度的整数倍。若每台设备持有 $v$ 个 chunk，则单 chunk 的前向/反向时间变为 $t_f/v$ 与 $t_b/v$，气泡时间变为

$$t_{pb}^{\text{int.}} = \frac{(p-1)\cdot(t_f + t_b)}{v}$$

对应气泡占比

$$\mathrm{BubbleFrac}^{\text{int.}} = \frac{1}{v}\cdot\frac{p-1}{m}$$

即气泡缩小 $v$ 倍；代价是 **点对点通信量也增加 $v$ 倍**。后续第 4 节会讨论怎样利用 DGX A100 的 8 张 IB 网卡吸收这笔额外通信开销。

### 2.3 张量并行

沿用 Megatron 的 Transformer 切分方式（见图 5）：Transformer 层由 self‑attention 块和一个两层 MLP 组成。

**MLP 块** 有两个 GEMM 与一个 GeLU：

$$Y = \text{GeLU}(XA),\qquad Z = \text{Dropout}(YB)$$

将 $A$ **按列** 切分 $A = [A_1, A_2]$，使 GeLU 可在各分片独立施加：

$$[Y_1, Y_2] = [\text{GeLU}(XA_1), \text{GeLU}(XA_2)]$$

这样就省掉了中间的同步（若按行切则因 GeLU 非线性而必须同步）。然后把 $B$ **按行** 切分：

$$B = \begin{bmatrix}B_1 \\ B_2\end{bmatrix},\quad Y = [Y_1, Y_2]$$

两个 GEMM 间无需通信，只在第二个 GEMM 输出后再做一次跨 GPU 的 all‑reduce（dropout 之前）。

**Self‑Attention 块** 利用多头的天然并行，把 $Q/K/V$ 矩阵按列分到多 GPU 上，输出的线性投影则按行分（见图 5b）。

图中的 $f, g$ 算子是共轭的：$f$ 在前向是恒等、反向是 all‑reduce；$g$ 反之。因此单层每前向/反向各有 **2 次 all‑reduce**。

## 3. 并行配置的性能分析

本节分析 $(p, t, d)$（流水并行度、张量并行度、数据并行度）在总 GPU 数 $n$ 与全局 batch 大小 $B$ 固定时的权衡。

### 3.1 记号

- $(p, t, d)$：三种并行度，满足 $p\cdot t\cdot d = n$
- $n$：GPU 总数
- $B$：全局 batch size（输入）
- $b$：micro‑batch size
- $m = \dfrac{1}{b}\cdot\dfrac{B}{d}$：单条 pipeline 上的 micro‑batch 数

### 3.2 张量并行 vs 流水并行

令 $d = 1$，则 $t\cdot p = n$。流水气泡占比为

$$\frac{p-1}{m} = \frac{n/t - 1}{m}$$

可见 $t$ 越大，气泡越小。通信方面：
- **流水并行**：相邻 stage 之间为点对点通信，每个 micro‑batch 的 P2P 通信量约为 $b\cdot s\cdot h$（$s$ 序列长度，$h$ 隐藏维）；
- **张量并行**：每层前向/反向各两次 all‑reduce，单微 batch 单设备的总通信量为

$$l_\text{stage}\cdot 8bsh\cdot\frac{t-1}{t}$$

张量并行显著更通信密集，所以一旦 $t$ 超过了单台服务器内 GPU 数（跨出高速 NVLink），就会遭遇跨节点链路的瓶颈。

> **Takeaway #1**：组合两类模型并行时，**张量并行度 $\le$ 服务器内 GPU 数 $g$**，之后用流水线并行跨服务器扩展。

### 3.3 数据并行 vs 两类模型并行

#### 3.3.1 流水 + 数据

令 $t=1$，$p = n/d$，则 $m = b'/d$，其中 $b' := B/b$。气泡大小为

$$\frac{p-1}{m} = \frac{n/d - 1}{b'/d} = \frac{n-d}{b'}$$

故 $d$ 越大、气泡越小；$B$ 增大（即 $b'$ 增大）同样减小气泡，并降低数据并行 all‑reduce 的频率。

> 图 6 给出了 $(d, n, b' = B/b)$ 组合下气泡占比的变化趋势。

注意受显存限制，不是所有模型都能把 $d$ 开到 $n$。

#### 3.3.2 数据 + 张量

张量并行的 all‑reduce 要在 **每个 micro‑batch** 上做（且大多数场景下跨机器都很昂贵）；而数据并行 all‑reduce 只每个 **batch** 一次。此外张量并行使每卡 GEMM 切得更小，可能跌出 GPU 的算术峰值。

> **Takeaway #2**：先选 $M = t\cdot p$ 的模型并行度让参数 + 中间元数据装下 GPU；再靠数据并行扩规模。

### 3.4 Micro‑batch 大小

micro‑batch $b$ 影响三件事：
1. 单个 GEMM 的算术强度，进而影响 GPU 实际算力；
2. 气泡大小（通过 $m = B/(b\cdot d)$ 间接影响）；
3. 激活显存。

只考虑计算时，一个 batch 的总计算时间可近似为：

$$T(b) = \Big(\frac{b'}{b} + p - 1\Big)\cdot\big(t_f(b) + t_b(b)\big)\tag{1}$$

> 图 7 给出 1B 参数 GPT 模型单卡吞吐 vs micro‑batch：更大的 $b$ 可带来最高 1.3× 的提升。
>
> 图 8 给出 $(p, t) = (8, 8)$ 时，按式 (1) 估计的归一化吞吐随 $b$ 的变化；在两种 batch 下最佳 $b$ 均为 4。

> **Takeaway #3**：最佳 $b$ 依赖模型吞吐/显存特性，以及 $p, d, B$。

### 3.5 激活重算（Activation Recomputation）

激活重算以多跑一次前向换取更低的激活显存占用：只保存每个 pipeline stage 的输入激活（远小于全部中间激活），在反向前重新跑前向得到中间激活。对于大模型上跑 pipeline 并行，激活重算几乎是必选项。

显存与 checkpoint 数量 $c$ 的关系：若每 stage 有 $l$ 层，输入激活大小 $A_\text{input}$、单层中间激活 $A_\text{intermediate}$，则总激活显存为

$$c\cdot A_\text{input} + \frac{l}{c}\cdot A_\text{intermediate}$$

取极小时 $c = \sqrt{l\cdot A_\text{intermediate}/A_\text{input}}$；经验上 **每 1~2 层做一次 checkpoint** 即可。

## 4. 实现

作者在 Megatron‑LM 代码库上用 PyTorch 实现，设备间通信使用 NCCL。下面是两类关键优化。

### 4.1 通信优化：scatter / gather

流水并行下前向/反向 P2P 通信会并行发生。每台 DGX A100 有 8 张 InfiniBand 网卡，但 send/recv 是点对点，单次调用难以用满 8 张卡。

利用一个关键观察：**Transformer 层输出（MLP 中 $g$ 之后）在张量并行 rank 间是复制一致的**。因此前后两个 pipeline stage 内做张量并行的 rank，互相发送的是完全相同的张量（见图 9a）。

若张量并行度为 8，就等于把同一张张量在相邻节点之间重复发送了 8 次。**scatter/gather 优化**：发送端把张量切成 8 等份，各 rank 用自己的 IB 网卡只发其中 1 块；接收端经 NVLink 做 all‑gather 把张量拼回来（见图 9b）。

量化：相邻 stage 间的总通信量由原先的 $b\cdot s\cdot h$ 下降到 $\dfrac{b\cdot s\cdot h}{t}$（实验中 $t = 8$）。该优化让交错式调度这种通信更密集的模式变得可行。

### 4.2 计算优化

三项针对 Transformer 的实现细节：
1. **改变张量布局**：把 `[b, s, a, h]` 改为 `[s, b, a, h]`（$a$ 为 head 数），避免内存密集型 transpose，并启用 strided batched GEMM；
2. **元素级 kernel 融合**：利用 PyTorch JIT 把 `bias + GeLU`、`bias + dropout + add` 等融合成一个 kernel；
3. **softmax 融合 kernel**：把 scale + mask + softmax 融成一个 kernel，分别为通用 mask（如 BERT）与隐式因果 mask（自回归 GPT）各写一份。

## 5. 评估

在 Selene 超算上做实验：每节点 8×A100(80GB) + NVLink/NVSwitch；每节点 8 张 NVIDIA Mellanox 200 Gbps HDR InfiniBand；三级胖树（leaf‑spine‑core）共 850 台交换机，NVMe 共享并行文件系统。A100 在 FP16 下理论峰值 312 TFLOP/s。

默认报告 **单卡吞吐**，聚合吞吐 = 单卡 × GPU 数。所有模型为 GPT，词表 $V = 51\,200$，序列长 $s = 2048$，参数量由 $h, a, l$ 决定：

$$P = 12 l h^2\Big(1 + \frac{13}{12 h} + \frac{V + s}{12 l h}\Big)\tag{2}$$

### 5.1 端到端性能（weak scaling）

> **表 1**：端到端训练可 **超线性** 扩展到 3072 张 A100：模型越大，GEMM 越大，GPU 利用率越高，而通信时间相对于计算并未显著增长。

主 GEMM（Transformer 层 + logit 头）的每次迭代 FLOP 可估为：

$$F = 96\,B s l h^2\Big(1 + \frac{s}{6h} + \frac{V}{16 l h}\Big)\tag{3}$$

（含激活重算中的额外一次前向）。

**训练时间估算**：给定 token 数 $T$，迭代数 $I = T/(B\cdot s)$。当 $6h \gg s$、$16lh \gg V + s$、$12lh \gg V$ 时，利用 (2)(3) 可近似得到：

$$\text{End-to-end training time} \approx \frac{8 T P}{n X}\tag{4}$$

其中 $X$ 是单卡实测吞吐。代入：GPT‑3（$P = 175\text{B}$、$T = 300\text{B}$ tokens、$n = 1024$、$X = 140$ TFLOP/s） ⇒ **约 34 天**；1T 参数模型（$T = 450\text{B}$、$n = 3072$、$X = 163$ TFLOP/s） ⇒ **约 84 天**。

### 5.2 与 ZeRO‑3 对比

在标准 GPT‑3 和 530B 模型上与 ZeRO‑3（单独使用，未配合模型并行）对比，见表 2 与图 10：

- 较少 GPU 且 $b = 4$ 时，PTD‑P 对 175B/530B 模型分别领先 6% / 24%；
- 将 GPU 数翻倍而 batch 不变时，PTD‑P 对两种模型都领先 ZeRO‑3 **约 70%**，主要因为跨节点通信更少；
- 作者指出这里仅是 "无模型并行" 的 ZeRO‑3 基线；ZeRO‑3 本身也可与张量并行组合。

### 5.3 流水线并行专项

#### 5.3.1 Weak scaling

用一个 $a = 128$、$h = 20480$、$b = 1$ 的 GPT 模型：当流水深度从 1 加到 8 时，对应把层数从 3（15B 参数）提到 24（121B 参数），$t = 8$ 保持不变，GPU 从 8 张到 64 张。

> 图 11 展示了两种 batch 下随 pipeline 级数变化的单卡吞吐。更大的 batch 由于把气泡摊到更多 micro‑batch 上，扩展性更好。

#### 5.3.2 交错式 vs 非交错调度

在 96 张 GPU 上的 GPT‑3（175B，96 层，96 头，$h = 12288$），图 12 显示：

- 带 scatter/gather 的 **交错式调度** 吞吐高于默认 1F1B；
- 随 batch 增大差距缩小，因为：(a) 默认调度的气泡在 batch 大时自然减小；(b) 流水内的 P2P 通信量随 batch 线性增长，而交错式每样本 P2P 更多。
- 若不开 scatter/gather，交错调度在大 batch 下反而可能被反超。

### 5.4 并行配置对比

#### 5.4.1 张量 vs 流水（图 13）

162.2B 参数 GPT 模型、64 张 A100，固定 GPU 数与 batch，扫描 $(t, p)$。结论：**张量并行在节点内最佳**（NVLink），流水并行用于跨节点扩展。流水阶段数也不能太多，要保留足够的 micro‑batch 数。最优点出现在 $t$ = 单机内 GPU 数（8）。

#### 5.4.2 流水 vs 数据（图 14）

5.9B GPT 模型（32 层、32 头、$h = 3840$），64 张 A100，$b = 1$。固定 batch 时，随 pipeline 级增大吞吐单调下降。**结论**：流水并行主要用于 "单卡装不下" 的情形，扩规模仍靠数据并行。

#### 5.4.3 张量 vs 数据（图 15）

同一 5.9B 模型。大 batch、小 micro‑batch 时，数据并行通信稀疏；而张量并行的同步要对每个 micro‑batch 做，跨节点尤其糟。张量并行度越大，单卡 GEMM 越小，利用率也下降。作者同时指出：对非常大的模型，纯靠数据并行是不可行的——既装不下，又会被 batch 大小限制所扼（GPT‑3 收敛时 batch=1536，纯 DP 至多 1536 卡，但实际用了约 10000 卡）。

### 5.5 Micro‑batch 大小

在 $(t, p) = (8, 8)$、91B 参数模型上扫描 micro‑batch，发现最优 $b = 2$。经验：$b$ 增大会减少 $m$、变大气泡，但也能增加算术强度。第 3 节的分析模型能合理近似真实吞吐，可作为调参参考。

### 5.6 激活重算

在 $(t, p) = (8, 16)$、128 张 A100、145B 参数（80 层、96 头、$h = 12288$）上对比：小 batch 时激活重算因多跑一次前向，序列吞吐最多降 33%；但它让 **大 batch 可行**，在大 batch 下启用激活重算能比最佳无重算吞吐高 **约 2×**（因气泡更小）。

### 5.7 scatter/gather 优化

175B GPT‑3、96 张 A100、交错调度下，scatter/gather 优化在通信密集场景（大 batch + 交错）带来最多 **11% 吞吐提升**。

### 5.8 算子融合

- 175B GPT‑3：融合后单卡 113 → 135 TFLOP/s（**+19%**）；
- 530B 模型：单卡 133 → 148 TFLOP/s（**+11%**）。

### 5.9 跨节点通信带宽

万亿参数 + 3072 GPU 上观测：
- 流水 P2P 的有效二分带宽 **892 GB/s**；
- 数据并行 all‑reduce 的有效二分带宽 **12.9 TB/s**。

切分不当会带来更多跨节点通信，扩展性受损。

### 5.10 checkpoint 加载与保存

万亿模型的 checkpoint **13.8 TB**。3072 卡（384 节点）初次加载可达 **1 TB/s 读带宽**（满速），保存达 **273 GB/s**（40% 写带宽峰值）。

## 6. 相关工作

**大模型并行**：流水并行方向上，TeraPipe 针对自回归模型沿序列 token 做细粒度流水；PipeTransformer 弹性调整流水与数据并行度，冻结 "稳定" 层；HetPipe 组合了异构加速器上的流水与数据并行。放宽语义的系列包括 PipeDream‑2BW（保留两份权重版本、1‑stale）、PipeMare 与 Kosson 等（异步流水）。这些方法吞吐更高但可能影响收敛；且纯流水并行可扩展设备数受层数限制。PipeDream 首次合理组合流水与数据并行；DeepSpeed 把三种并行组合到万亿参数规模，但吞吐为峰值 36%（低于本文 52%），原因在于本文用了更优的算子融合、更高效的流水调度、更快的硬件（A100 vs V100）以及更大规模。Mesh‑TensorFlow 提供了声明式并行语法；Switch Transformers 用其把 MoE 扩到 1.6T 参数。

**分片数据并行**：MLPerf 0.6 的分片优化器状态、ZeRO‑1/2/3 把优化器状态/梯度/参数逐级切分到 DP worker，再通过通信重组相关状态；ZeRO‑Infinity 更是借 NVMe 交换参数。缺点是这类方案在未配合张量并行或 batch 不够大时，额外通信难以被隐藏（见图 10）。

**自动切分**：FlexFlow、PipeDream、DAPPLE、Tarnawski et al. 等用代价模型自动搜索切分；但没有任一工作同时覆盖本文关注的全部维度（流水 + 张量 + 数据 + micro‑batch + 激活重算）。Gholami 等给出了数据/模型并行组合下的通信代价建模。

**HPC for 训练**：Goyal、You 等在 ImageNet 分钟级训练上的成果不适用于本场景，因为 ImageNet 模型装得下单卡、batch 可开到 32k+、卷积本身亲和数据并行。

## 7. 讨论与结论

作者展示了 PTD‑P（节点间流水 + 节点内张量 + 数据并行）可组合出 **502 PFLOP/s** 的聚合吞吐、在万亿参数模型上把端到端训练时间压到约 **3 个月**。各类并行的代价与相互作用被系统性地分析过。

作者强调，即便实现是 GPU 中心的，以下思想对其他加速器仍然通用：
- **智能切分计算图** 以降低通信、同时保持设备忙碌；
- **算子融合** 与 **合理数据布局** 降低显存瓶颈 kernel；
- 其他领域特定优化，例如 **scatter/gather**。

## 附录：FLOP 数如何计算

设 $l$ 层 Transformer、隐藏维 $h$、序列长 $s$、词表 $V$、batch $B$。

- 一次 $A_{m\times k}\times X_{k\times n}$ 矩阵乘消耗 $2mkn$ FLOP（乘 2 记乘加）。
- Attention 块的主要 FLOP：
  - $Q, K, V$ 投影 $6 B s h^2$
  - 注意力矩阵 $2 B s^2 h$
  - Attention 乘 $V$ $2 B s^2 h$
  - 后投影 $2 B s h^2$
- FFN：先升到 $4h$ 再降回 $h$，共 $16 B s h^2$。

合计单层 **前向** FLOP：

$$24 B s h^2 + 4 B s^2 h$$

**反向** 是前向的 2× 倍（对输入与权重都要求梯度），再加上激活重算的 **额外一次前向**，所以总 FLOP 为 $4\times$ 前向：

$$4\times(24 B s h^2 + 4 B s^2 h) = 96 B s h^2\Big(1 + \frac{s}{6h}\Big)$$

logit 层将 $h$ 维特征映射到词表 $V$，前向 $2 B s h V$、反向 $4 B s h V$，共 $6 B s h V$。

因此 $l$ 层 Transformer 总 FLOP：

$$96 B s l h^2\Big(1 + \frac{s}{6h} + \frac{V}{16 l h}\Big)$$
